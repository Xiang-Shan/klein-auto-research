The Data

The dataset is the Kaggle litvinenko630/insurance-claims file: 58,592 policies, 41 raw columns (45 after prepare.py), with a binary target claim_status that is 6.40% positive (3,748 claims). Those three numbers — row count, prepared column count, and positive rate — match the campaign's own documented figures exactly, which is the first data-integrity signal, before the modeling even starts.

The war story that gate 1 exists to prevent

Here is the trap, and it is not a museum piece — it reproduced live in this environment. Nineteen columns in this data are really booleans wearing string clothes: the seventeen is_* feature flags hold exactly "Yes" / "No", and two more (rear_brakes_type, transmission_type) hold "Disc"/"Drum" and "Automatic"/"Manual". The obvious way to find categorical columns is to check the dtype:

# The bug. Looks reasonable. Silently wrong.
categorical = [c for c in df.columns if df[c].dtype == "object"]

Under this machine's pandas 3.0.3, every one of those nineteen columns reports its dtype as str — not "object". Pandas 3.0 made a dedicated string type the default storage, so the label changed. The check above matches none of them, silently skips all nineteen boolean features, and hands the model a pile of raw "Yes"/"No" text. In the ancestor campaign this cost about two hours of confusion before anyone noticed the features had gone missing.

The fix is a discipline, not a patch: never key on the dtype label; inspect the actual values.

# The fix. Detect by VALUE PATTERN, not by dtype.
def detect_yes_no_columns(df):
    out = []
    for col in df.columns:
        vals = set(df[col].dropna().unique())
        if vals <= {"Yes", "No"}:          # membership on the real values
            out.append(col)
    return out

yes_no_cols = detect_yes_no_columns(df)     # finds all 17, dtype label be damned
for col in yes_no_cols:
    df[col] = df[col].map({"No": 0, "Yes": 1}).astype("int64")

After conversion, prepare.py verifies all nineteen are int64 with zero nulls across the 58,592 rows. The same principle handles the numbers-in-strings columns (max_torque = "113Nm@4400rpm"), which are regex-parsed into clean numeric columns rather than treated as free text. This value-pattern discipline is exactly why the E1 anchor could reproduce the campaign's prepared frame bit-for-bit.

The go / no-go call

Gate 1 ranks every issue and blocks modeling until the verdict is GO. Here there were three WARN-severity issues and no blockers:

  • WARN — the 19 string-boolean columns above. Mitigated by the value-pattern conversion; verified int64 post-fix.
  • WARN — seven vehicle-spec columns (engine_type, displacement, cylinder, and four parsed max_torque_*/max_power_* columns) are near-deterministic functions of model — redundant, and a source of collinearity for a GLM. Kept for the LR anchors (to reproduce the campaign exactly), dropped for the HGBT recipe.
  • WARN — the 6.4% class imbalance. Handled not by re-weighting but by the calibration doctrine (Section 2), which RQ2 puts directly to the test.

No leakage column, no broken split, no unusable encoding once the value-pattern conversions are applied. Verdict: GO. The full audit is in data_card.md.
