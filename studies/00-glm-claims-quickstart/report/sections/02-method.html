The Method

The workhorse here is logistic regression — and if you price with GLMs, you already know it. It is the same machinery as your Poisson-frequency or Gamma-severity models, just with a Bernoulli response and a logit link, aimed at P(claim) instead of a count or an amount. No new mathematics; the skill is in how you feed it.

The linear predictor, and why it is both blessing and curse

The model scores each policy on the log-odds scale with a plain weighted sum:

η = β₀ + β₁·x₁ + β₂·x₂ + … + βₖ·xₖ      (the linear predictor)
P(claim) = 1 / (1 + e^(−η))                (the logit link squashes it into [0,1])

Each coefficient β₃ is a log odds-ratio: holding everything else fixed, a one-unit move in x₃ multiplies the odds of a claim by e^β₃. An underwriter — or a regulator — can read the coefficient table straight off. That transparency is the GLM's blessing.

The curse is the flip side of the same coin. Because the score is a straight sum, the model is monotone and additive by construction: it literally cannot see a U-shape, a threshold, or an interaction between two features unless you build that shape into the design matrix yourself. A boosted tree discovers curvature and interactions on its own; a GLM never will.

Why splines — the model's only path to a curve

So we hand the model its non-linearity. The standard tool is a spline basis: instead of feeding customer_age as one column, you expand it into a handful of smooth basis functions, and the GLM fits a separate weight to each. The sum of those weighted pieces is a smooth curve in age — risk that rises, plateaus, and falls — which a single linear term could never express. Splines on the two or three most important continuous features are mandatory feature engineering for a linear model on this data (worth about +0.026 AUC in this study; see the journey). This is the exact opposite of the rule for trees, where hand-built features mostly just add noise.

Two details decide whether splines actually pay off, and neither is captured by the phrase “5 knots” — they are the traps this study fell into and climbed out of (Section 6):

  • Where you place the knots. knots="quantile" puts them where the data actually lives; the sklearn default "uniform" spaces them evenly and wastes resolution on empty ranges.
  • Whether you include a bias column. include_bias=False, because the regression already has its own intercept — the default duplicates it and quietly degrades the fit.

Why calibration comes first on a 6.4%-positive target

Here is the idea that trips up most newcomers: ranking and calibration are not the same thing. AUC only asks “did the model put riskier policies above safer ones?” A model can rank acceptably and still be badly biased in the actual probability it reports — and for pricing, the probability is the product.

The instinct on an imbalanced target is to re-weight the loss so the rare positives “count more” (class_weight="balanced"). Do not. Re-weighting barely moves AUC and it systematically inflates the predicted probabilities — the model now thinks claims are far more common than 6.4%. The campaign found this across every re-weighting trick it tried, and this study reproduces it live (Section 5). The right recipe:

  1. Train on the natural distribution: class_weight=None. Don't lie to the model about the base rate.
  2. Fix the probabilities afterward with isotonic regression (CalibratedClassifierCV(method="isotonic", cv=5)) — a monotone step function that re-maps raw scores onto the observed claim frequencies. It preserves the ranking and repairs the bias. This is the actuarially load-bearing step.
  3. Tune the decision threshold last, for whatever operating point the business needs.

Put together, the whole model is a short pipeline — the encoder, spline, and calibration choices are the experiment; everything else is library:

from sklearn.preprocessing import OneHotEncoder, SplineTransformer
from sklearn.linear_model import LogisticRegression
from sklearn.calibration import CalibratedClassifierCV

# OHE is non-negotiable for nominal categories in a linear model (never ordinal):
#   OHE 0.6255  vs  ordinal 0.4039  -- a wrong encoder makes LR unusable.
pre = ColumnTransformer([
    ("cat", OneHotEncoder(min_frequency=20, handle_unknown="infrequent_if_exist"), cat_cols),
    ("num", SplineTransformer(n_knots=5, degree=3, knots="quantile", include_bias=False), spline_cols),
])
glm = Pipeline([("pre", pre), ("lr", LogisticRegression(C=1.0, class_weight=None, max_iter=2000))])
model = CalibratedClassifierCV(glm, method="isotonic", cv=5).fit(X_tr, y_tr)

One honest caveat, so you set expectations correctly: the signal in this dataset is weak. The score histograms for the two classes overlap almost completely — the best any linear model reaches is val_auc ≈ 0.65, and a tuned boosted tree only gets to ~0.66. There is no configuration that makes this an easy separation problem; the GLM's job is a transparent, well-calibrated anchor, not a leaderboard win.
